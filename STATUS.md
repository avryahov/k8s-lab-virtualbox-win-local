Этот файл нужен как живая точка синхронизации для пользователя и любого следующего ассистента.

Здесь фиксируются:

## Главная цель проекта

Проект должен быть одновременно:
- рабочим учебным Kubernetes-стендом;
- понятным учебным материалом для школьников и начинающих.

Это означает два обязательных требования:
- кластер должен стабильно подниматься и работать;
- Vagrantfile, shell-скрипты, PowerShell-скрипты и документы должны объяснять, что происходит и зачем.

## Согласованная логика этапов stage1

Текущая согласованная учебная логика такая:

1. Поднять 3 ноды.
2. Выполнить `kubeadm init` на master.
3. Выполнить `kubeadm join` для worker-нод.
4. Настроить Pod-сеть и Calico.
5. Прогнать smoke-тест простого приложения.
6. Только после этого установить Dashboard.

Причина:
- учебно это понятнее;
- технически это снижает вероятность того, что второстепенные сервисы сломают основной bootstrap;
- сначала нужно доказать, что работает сам кластер, и только потом добавлять удобства.

## Что ещё не завершено

### Чистый повторный прогон

Нужно:
1. Выполнить `vagrant destroy -f` в `stage1`.
2. Проверить хвосты VirtualBox и убедиться, что чужие кластеры не затронуты.
3. Поднять `stage1` с нуля.
4. Повторно выполнить пост-финализацию.
5. Ещё раз подтвердить:
   - 3 ноды `Ready`;
   - Calico стабилен;
   - smoke-тест проходит;
   - Dashboard доступен в самом конце.

### Учебная финализация

После чистого прогона нужно:
1. Проверить, что новые комментарии действительно соответствуют фактическому порядку шагов.
2. При необходимости дополнительно усилить пояснения в учебных материалах.

## Глобальные правила, которые уже надо помнить

### Язык

Все ответы пользователю — на русском языке, кроме:
- кода;
- CLI-команд;
- путей;
- логов;
- точных идентификаторов;
- точных цитат.

### Запрет на служебные названия

Нельзя использовать названия ассистентов, моделей или нейросетей:
- в названиях веток;
- в описаниях;
- в комментариях;
- в документах;
- в инструкциях;
- в постоянной памяти.

Это правило уже сохранено глобально для всего ПК.

### Git-flow пользователя

Обязательная схема:
- `main`
- `develop`
- `feature/*`
- `bugfix/*`
- `hotfix/*`

Дополнительно:
- не более 6 файлов в коммите;
- не более 20 коммитов на ветке;
- минимум 3 коммита на ветке перед дальнейшим движением;
- сообщения коммитов только на английском;
- merge только через `--no-ff`;
- никаких destructive git-операций без явного разрешения пользователя.

### Правила PowerShell для этого проекта

Нужно помнить и соблюдать:
- host-side команды надо запускать из правильного каталога, для `stage1` это `stage1`, а не корень репозитория;
- в Windows PowerShell 5 нельзя использовать `&&`, нужно использовать `;` или отдельные команды;
- вложенные кавычки в `vagrant ssh ... -c ...` надо упрощать и по возможности использовать одинарные кавычки снаружи;
- для сложных host-side сценариев лучше выносить логику в `.ps1`, а не собирать длинные inline-команды;
- внешние утилиты вроде `VBoxManage` могут писать в `stderr` даже в ожидаемых сценариях, поэтому их нельзя обрабатывать наивным вызовом без проверки `ExitCode`;
- при работе с `vagrant ssh` и PowerShell нужно заранее фиксировать `workdir`, чтобы Vagrant видел именно нужный `Vagrantfile`.
